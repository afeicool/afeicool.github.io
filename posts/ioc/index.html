控制反转（Inversion of Control, IoC）浅析

September 1, 2019 2 minutes read
IoC • DI • 控制反转 • 信赖注入

控制反转（Inversion of Control, IoC）的概念

IoC 是一种程序设计的理念，它主要解决的是程序耦合的问题。而实现这个理念有多种 方式，而最常见的就是 依赖注入 。

IoC 解决了什么问题

首先让我们看一个最常见的例子： 在订单生成时，抛一个事件出去： 

  // 示例1

  class Order {
      private final EventPublisher eventPublisher;

      public Order() {
          eventPublisher = new EventPublisher();
      }

      public long save () {
          //...

          eventPublisher.publishEvent("save");

          //...
      }
  }

这里略无关业务逻辑的代码，聚焦在 Order 和 EventPublisher 关系上， Order 是依赖于 EventPublisher 的，于是 Order 在构造时，自己把 EventPublisher 创建出来了。

问题是 Order 需要知道 EventPublisher 如何创建的，这好比某人想吃猪肉，于是 他自己养了猪，自己磨了把杀猪刀是一个道理。看到 new Object , 基本也就断定了， 产生了紧耦合。换句话说 Order 需要知道它本并不需要知道的事，它知道太多了。

这里违返了两个设计原则，第一是 开闭原则 ，第二是 针对接口编程 。我们需要把 可能的变化屏蔽或封装起来，这里的变化是EventPublisher本身的实现是可能有多种的， 比如同步实现、异步实现、通过消息中间件的实现，等等， Order 本身不关心这是什 么实现，只需要知道需要触发时，调用接口就好了，剩下的事，由具体实现去关心。

因为绑定了具体实现，同样导致测试不好测，这里没办法在测试里Mock一个 EventPublisher 。代码难以测试，也表明代码设计是有问题的。

依赖注入 (DI)

依赖注入是 IoC 实现的方式之一，另外的实现还有服务查找(Service Locator)等， 这里只说依赖注入。

重构一下上面的例子：

构造函数注入 

  // 示例2: 构造函数注入

  class Order {
      private final IEventPublisher eventPublisher;

      public Order(IEventPublisher eventpublisher) {
          this.eventPublisher = eventpublisher;
      }

      public long save () {
          //...

          this.eventPublisher.publishEvent("save");

          //...
      }
  }

Order 现在不管 EventPublisher 如何实例化，反过来由 外部 供给, 或者称之 为 注入 。并且这里传入的只要是 IEventPublisher 这个接口的任意实现。所谓 的 控制反转 ， 就是指依赖的提供方由内而外，不自己提供，反过来由 外部 提 供。这里的 外部 一般称为 容器 (Container) 或 提供方(Provider) 。很多语 言框架都提供了自己的容器实现，比如php的框架 Laravel 、Java 框架 Spring (Spring 容器的实现，会在Spring 系列文章来详述。)

上述例子(示例2)中，依赖是通过构造器来注入的，所以称之为 构造器注入 。

属性注入

同样的，还有通过属性的设置来注入： 

   // 示例3: 属性注入

   class Order {
       private final IEventPublisher eventPublisher;

       public setEventPublisher(IEventPublisher eventPublisher) {
           this.eventPublisher = eventPublisher
       }

       public long save () {
           //...

           this.eventPublisher.publishEvent("save");

           //...
       }
   }

上述例子(示例3)中，依赖是通过属性设置来注入的，所以称之为 属性注入 或 Setter注入 。

工厂方法注入

在Java中，对象实例化的方式除了使用 new 以外，用得比较多的算是 工厂方法 ，所以还可以通过工厂方法来实现注入： 

   // 示例4: 工厂方法注入

   class Order {
       private final IEventPublisher eventPublisher;

       private setEventPublisher(IEventPublisher eventPublisher) {
           this.eventPublisher = eventPublisher
       }

       public static Order OrderFactoryMethod(IEventPublisher eventPublisher) {
           Order order = new Order();
           order.setEventPublisher(eventPublisher);
           return order;
       }

       public long save () {
           //...

           this.eventPublisher.publishEvent("save");

           //...
       }
   }

上述例子(示例4)中，依赖是通过静态工厂方法来注入的，所以称之为 工厂方法注入 或 Factory Method 注入 。

比较

构造器注入 一般都是强约束的，所以一般强依赖时，使用构造器注入。此外，像 Spring框架中，构造器注入是可以避免循环依赖，这一般是一个比较好的编程实践，而 属性注入 是允许循环依赖的。

属性注入 一般都是可选的，所以一般可选依赖时，使用属性注入。

工厂方法注入 在需要使用工厂模式时，工厂方法注入一般都是比较合适的。

总结

总体来说 控制反转 及其实现之一 依赖注入 名字乍看比较费解，但了解了其背后解 决的问题，和简单的实现之后，还是很好理解的。非常简单的概念,但是极其强大，现代 框架里，很多都是基于这一简单的概念为核心，构建和组织复杂的面向对象框架的。
